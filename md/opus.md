# quickquery 相談メモ（Opus）

---

## 質問 1

@quickquery この自作ライブラリを読み、コンセプトを逸脱しない範囲で、改善や機能追加案などを考えてみて。要は、今はない、「こういうのどう？」というアイデアが欲しい。いいのがあれば。コンセプトのアイデアでもいいし、なんでもOK

---

## 回答 1

quickquery を一通り読みました。まず「逸脱しない範囲」を判断する基準として、私が読み取ったコンセプトを言語化しておきます。ここがズレてたら指摘してください。

## このライブラリのコンセプト（私の理解）

- **極薄ラッパ + 徹底した null 安全チェーン**: `i`/`ii`/`n`/`p`/`o`/`scan.m` が全部「壊れない」。`None` でも例外を投げず空ラッパを返し、`__bool__` で分岐できる。`if not p.goto(...)` のスタイルが全体で一貫している。
- **live と parse の API 対称性**: `QuickPage/QuickElement`（Playwright）と `QuickParser/QuickNode`（selectolax）が同じ語彙（`i`/`ii`/`n`/`p`/`o`/`scan`）を共有。
- **2段構えのワークフロー**: 「生HTMLを meta 付きで丸ごと保存（真実の記録）」→「後から並列で何度でも再抽出」。`hash_name` でURL単位に冪等、`meta_html` で来歴を埋め込む。
- **自分用の実用主義**: loguru で握りつぶしログ、`RecycleEvery` で長時間クロール、Excel互換BOM など「現場で効く」細部。

この4本柱を壊さない範囲で、「今ない」アイデアを強い順に挙げます。

---

## A. 対称性の穴を埋める（コンセプト直球・低リスク）

### 1. `QuickNode` に `url` / `src`（相対URL解決）を生やす ★イチオシ

今 `QuickElement` には `urljoin(page.url, ...)` する `url`/`src` がありますが、**`QuickNode` 側には無い**。だから抽出フェーズで `<a href="/foo">` を絶対URLに直せません。

しかも面白いことに、**あなたは既に `meta` に `quickquery:final_url` を保存している**。つまり parse 側でも base URL は手元にある。これを使えば対称性が完成します。

```python
# proposed
def quick_parser(parser, base_url: str | None = None) -> QuickParser: ...

# QuickNode に
@property
def url(self) -> str | None:  # href を base_url で urljoin、javascript:/# 等は除外
    ...
```

`base_url` は `read_meta`（下の#5）から自動で拾えると最高に噛み合います。

### 2. `.html`（inner/outer HTML）プロパティ ★

抽出が空振りしたとき「その要素の中身どうなってた？」を見るのに、`QuickElement`/`QuickNode` 両方に `.html` があると数万件のデバッグが激減します。selectolax は `node.html`、Playwright は `inner_html()` で対称に実装可。

### 3. グループの `.one` / `.first`

`ii(...)[0]` は「想定外に複数/ゼロ」を黙って通します。`.one`（1個でなければ warning ログ）は、サイト構造変化に伴うサイレント崩れの早期検知になり、まさに「null安全＋ログ」の思想の延長です。

```python
elem = p.ii('h1.price').one  # 1個以外なら logger.warning して [0] or None
```

---

## B. ワークフロー堅牢化（2段構えコンセプトの自然な延長）

### 4. マニフェスト＋再開（resume）★イチオシ

今の `scrape.py` は**全URLを毎回取り直し**、来歴は各HTMLの `<meta>` を読み直して復元する作りです。スクレイプ中に落ちたら最初からになります。

`scrape` 時に1行ずつ `index.csv`（or parquet）へ `url_index, request_url, final_url, hash_name, saved_at, status` を追記すれば：

- **再開**: 既にある `hash_name` をスキップ（`already(path)` ヘルパ or マニフェスト参照）
- **突合**: 抽出結果と来歴の付き合わせが一発
- `failed.csv` と対になる「成功台帳」になる

これは新概念ではなく、`meta_html`/`hash_name` が既に持っている思想を1ファイルに集約するだけなので逸脱しません。

### 5. `read_meta(parser, prefix='quickquery:')` ★

`meta_html` で**書く**ヘルパはあるのに**読む**ヘルパが無く、抽出側で4回手書きしてます：

```python
'url_index': p.i('meta[name="quickquery:url_index"]').attr('content'),
'saved_at':  p.i('meta[name="quickquery:saved_at"]').attr('content'),
...
```

`meta_html` の対になる `read_meta` を足すと、書き/読みが対称になり #1 の `base_url` 供給源にもなります。

```python
meta = read_meta(parser)  # {'url_index': ..., 'final_url': ...}
p = quick_parser(parser, base_url=meta.get('final_url'))
```

### 6. `bytes_at` を `context.request.get` ベースに

現状 `bytes_at` は**新規ページを開いてナビゲート**してバイトを取りますが、Playwright の `page.context.request.get(url)` なら**同じ cookie/セッションを共有したHTTPフェッチ**で済み、ページ生成オーバーヘッドが無く速い・安定。画像取得（`media/*.jpg`）の用途に直結します。挙動が変わるので任意切替でも可。

### 7. 圧縮HTML（`.html.gz`）透過対応

「生HTMLを真実として全部残す」コンセプトだと数万件でディスクが効いてきます。`write_text`/`parse_html` が拡張子で gzip を透過処理できると、コンセプト（保存して再抽出）を維持したまま容量を大きく削れます。selectolax は bytes を食えるので展開もシンプル。

---

## C. コンセプト寄りの細かいアイデア

- **8. `goto` の HTTP ステータス認識**: 今は response が非Noneなら 404/500 でも「成功」として保存対象になります。`goto(..., expect_ok=False)` のような任意フラグで「4xx/5xx はリトライ or 失敗扱い」を選べると、`failed.csv` の精度が上がります（デフォルト現状維持なら非破壊）。
- **9. `scan` の属性版**: `scan.m` はテキスト正規表現専用。`href`/`alt` 等の属性に対する正規表現マッチ（`scan_attr('href').m(...)`）があると、テキストが空のリンク・画像の選別に効きます。`ii('a[href^=...]')` で足りる場面も多いので優先度は中。
- **10. 型の Literal 化**: `w(..., state: str)` は `Literal['attached','visible','hidden','detached']`、`__getitem__` の返り値 union など。実害は薄いですが、`from __future__ import annotations` + 全部型付けの今の丁寧さと揃います。

---

## あえて「却下」した案（コンセプト逸脱）

- **宣言的スクレイプ定義（YAML/dictでセレクタ宣言→自動抽出）**: 一見便利ですが、`extract.py` の「Pythonで明示的に書く」「正規表現で泥臭く拾う」現在のスタイルと真逆。魔法を増やす方向は非汎用・実用主義のコンセプトから外れると判断。
- **async対応**: sync で一貫した潔さが今の魅力。混ぜると対称性が崩れる。

## 回答 2

いいですね、機能じゃなくて「思想」の話。コードを読んでいて感じた**設計上の緊張点（tension）**を軸に、踏み込んだ提案をします。機能追加というより「このライブラリは何を信じているのか」を一段クリアにする話です。

---

## 1. 一番大事だと思う点：「失敗の握り方」を2層に分ける ★本命

今このライブラリは **全レイヤーが fail-soft（握りつぶしてログ＋継続）** で統一されています。`goto` のリトライ、`scan.m` の空ラッパ、`text` が `''→None`、`process_map` の `None` 返し… 全部「壊れず流す」。

これはクロールでは**正解**です。ネットワークやブラウザは本質的に不安定で、1件のために全体を止めたくない。

でも同じ姿勢が**抽出（意味）レイヤー**にも効いているのが、思想的な弱点だと思います。

```python
'価格': dd_text(r'価格'),   # サイト改修で取れなくなっても、静かに None
```

数万件回したあと、`価格` が3割欠損していても**誰も気づかない**。気づくのは数ヶ月後の分析時。これは「データの静かな損失」で、クロールの一時失敗よりずっと怖い。

**思想として提案したいのは「2つの世界を分ける」こと:**

- **トランスポート層（network/browser）= fail-soft**：今のまま。落ちても流す、リトライする。
- **意味層（extraction）= fail-visible**：壊れたら**見える**べき。例外で止めろとは言わないが、「期待」と「実績」を記録すべき。

具体的には *expectation / coverage* という概念を一つ持つ。抽出後に「`価格` は98%取れた、`間取` は40%しか取れてない」というレポートが自動で出る。`failed.csv` が「取得の失敗台帳」なら、これは「**理解の失敗台帳**」。今のコンセプト（ログで握る・台帳に書く）の自然な拡張で、新しい魔法ではありません。

> 一言で：**「取れなかった」と「取れたけど中身が変」を区別する思想がまだ無い。**

---

## 2. スクレイプは「現在の状態」じゃなく「観測の記録」だと宣言する

あなたは既に `saved_at` / `request_url` / `final_url` を meta に埋めています。これは無意識に**正しい思想**に足を踏み入れている証拠です。

ここを思想として言い切ると強い：**「このライブラリが作るのはスナップショットではなく、時刻つきの観測アーカイブ（observatory）である」**。

- 1件の事実は「真実」ではなく「**時刻Tに源SからこういうHTMLが観測された**」という主張。
- だから同じURLを再スクレイプしても**上書きで消さない**（今は `hash_name(url)` で上書き）。`hash_name(url + saved_at)` 的にして履歴を残せば、「価格がいつ変わったか」「物件がいつ消えたか」という**時系列データ**が無料で手に入る。

不動産データなんてまさに「いつの時点か」が命なので、この思想転換は実利が大きい。`hash_name` の設計一つに「上書き=現在主義」か「追記=観測主義」かという哲学が宿っています。

---

## 3. 「生HTML＝唯一の真実、それ以外は再生成可能な射影」を徹底する

`crawl → scrape(生HTML保存) → extract → clean` の流れは、実は**関数型のデータパイプライン**です：

- `extract` は理想的には **純関数** `(html) -> dict`。`process_map` で並列できるのもこれが純粋だから。
- 下流（parquet, clean）は全部「生HTMLから再生成できる射影」にすぎない。

これを思想として明示すると、一つ強い帰結が出ます：**抽出ロジックにバージョンを持たせる**。

```python
# meta か台帳に
'extractor_version': 'v3',
```

そうすると「抽出ロジックを直した → v2 と v3 の結果を **diff** できる」「再生成が決定的（同じ生HTML＋同じバージョン＝同じ結果）」になる。生HTMLを真実として貯め込む今の設計は、**再現性のあるパイプライン**まであと一歩。`hash_name` で冪等にしている冪等性思想の、抽出側への延長です。

---

## 4. live と parse の「対称性」を“手作業の慣習”から“契約”へ昇格する

前回 `QuickNode` に `url` が無かった件は、対称性が**規律ではなく善意で保たれている**証拠でした。だから片肺が生える。

思想として「**live と parse は置換可能でなければならない（Liskov）**」を掲げ、共有 `Protocol`（`i/ii/n/p/o/text/attr/url...`）を一枚定義する。すると：

- 対称性の崩れが**型エラーとして検出**される（善意に頼らない）。
- さらに大きいのは、**抽出ロジックを live/parse 両対応で1回だけ書ける**こと。`extract(p)` が `QuickPage` でも `QuickParser` でも動く。デバッグ時はライブで試し、本番は保存HTMLを並列パース、という往復が同じコードでできる。

「同じ語彙」を duck typing の偶然ではなく**設計の法則**にする、という話です。

---

## 5. 「人間に見える文字を錨にする」を第一級の原則に

地味ですが、`dt_scan.m(r'価格').n('dd')` という書き方は**思想的にすごく良い**。class名やDOM構造（＝機械の都合、改修で消える）ではなく、**人間が読むラベル（価格・所在地…これは滅多に変わらない）を錨にしている**。`scan` が NFKC 正規化してるのも「人間の見た目で揃える」思想と一貫。

これを「たまたまそう書いた」で終わらせず、**ライブラリの公式スタンス**として言語化・強化する価値があります。「**markup は揮発性、テキストは安定**だから、原則テキストアンカーで取る」。`dt/dd`・`th/td`・`label/値` のラベル起点抽出を第一級にすると、サイト改修への耐性が構造的に上がる。`scan` はその思想の芽。

---

## 6. 空（None）に「なぜ空か」を記憶させる ─ Maybe with provenance

今の null 安全は美しいけど、**理由を捨てている**。`p.i('a').n('dd').text` が None のとき、`a` が無かったのか `dd` が無かったのか分からない。数万件のデバッグでは、この「**どこで途切れたか**」こそが知りたい情報。

思想：空ラッパに「**どのセレクタまで生きてて、どこで死んだか**」のパンくずを持たせる。例外は投げない（fail-soft 思想は維持）が、`elem.trail` で「`a` は取れたが `n('dd')` で消滅」と言える。`__bool__` で分岐する今のスタイルを壊さず、**Maybe を Either に近づける**だけ。#1 の coverage とも噛み合います。

---

## 7. 「自分用」を制約ではなく“武器”として規律化する

最後にメタな話。`patchright + playwright + camoufox` と既にバックエンド3つ。汎用ライブラリへ静かに育ちつつあります。

「自分用・非汎用」は**言い訳ではなく設計上の強み**です。汎用ライブラリが取れない「雑だけど速い・壊れても自分が直せる・自分のジョブに最適化」という選択ができる。だからこそ思想として **anti-goals（やらないことリスト）** を README に明記する価値がある：

- 宣言的DSLを作らない（imperative で上から読める潔さを守る）
- async を足さない（sync の一貫性を守る）
- 「他人が使える汎用化」を目的にしない（追加基準は「**自分が実ジョブで要るか**」だけ）

メス（scalpel）であり続けるための規律。機能を足すより、**足さない理由を言語化する**方が、このライブラリの思想を守ります。

---

## まとめ（回答 2）

私の主観での重要度は **1（失敗の2層化）＞ 2（観測アーカイブ）＞ 3（再現可能な射影）**。この3つは互いに補強し合い、しかも全部「あなたが既に半分やっていること（meta, hash_name, failed.csv, 純関数なextract）」の延長線上にあるので、**思想を変えるというより“言い切る”だけ**で芯が通ります。

どれか掘り下げます？ あるいは「自分はこう思ってた」という反論があれば、そこを一緒に詰めるのが一番面白いと思います。
